1. EXECUTIVE SUMMARY
================================================================================

The two packages serve COMPLEMENTARY but OVERLAPPING purposes:

  custom-requests  = Employee request lifecycle (submit -> approve -> fulfill)
  store-operations = Document-driven inventory accounting (receipts, issues, ledger)

CRITICAL FINDING: The Fulfillment workflow in custom-requests updates Snipe-IT
quantities DIRECTLY without creating any audit document in store-operations.
This creates a gap where stock leaves the building but no Goods Issue document
exists in the immutable ledger.

Additionally, store-operations has its own Goods Issue flow that duplicates what
Fulfillment already does, risking user confusion about which UI to use.

3. DETAILED WORKFLOW ANALYSIS - custom-requests
================================================================================

3.1 Request Submission (Employee)
---------------------------------
  Path: /gov-requests/catalog or /gov-requests/basket
  
  Two submission methods exist:
  
  A) Single-item request (ItemRequest model - legacy/simple):
     - User clicks "request" on any item page
     - Creates a custom_item_requests record
     - Status: pending
     - NO approval workflow integration
   
  B) Multi-item service request (Request model - primary):
     - User browses catalog or uses basket
     - Groups items by approval policy
     - Creates custom_service_requests record(s)
     - Auto-approval if policy allows, else routes to approver

3.2 Approval Processing (Approver)
----------------------------------
  Path: /gov-requests/admin
  
  Three approval policies:
  - AUTO_APPROVE:   System auto-approves, no human needed
  - PRIMARY_ONLY:   Single approver level
  - PRIMARY_AND_FINAL: Two-level approval chain
  
  Approver actions:
  - Approve with optional quantity modification
  - Reject with reason
  - Partial approval (some lines approved, some rejected)
  
  Self-approval conflict detection:
  - If requester is also final_approver in PRIMARY_AND_FINAL, auto-approves

3.3 Fulfillment (Storekeeper)
-----------------------------
  Path: /gov-requests/fulfillment
  
  Storekeeper actions:
  1. View approved requests queue
  2. Open request detail
  3. Issue items with optional product substitution
  4. Partial fulfillment supported
  5. Close request when complete
  
  CRITICAL: Fulfillment calls RequestableFactory->checkout() which directly
  updates Snipe-IT tables via Snipe-IT's native relationships. NO Goods Issue
  document is created in store-operations.

3.4 Key Models
--------------
  ItemRequest       -> Single-item, simple pending status (legacy)
  Request           -> Multi-item service request with full lifecycle
  RequestItem       -> Line items with approval/fulfillment tracking
  RequestEvent      -> Timeline/events log
  ApprovalPolicy    -> Category-level policy configuration
  LocationRole      -> User role assignments per location

4. DETAILED WORKFLOW ANALYSIS - store-operations
================================================================================

4.1 Goods Receipt (Inbound Stock)
----------------------------------
  Path: /gov-store/operations/receipts/create
  
  Storekeeper receives goods, fills Fast Grid, submits.
  Creates: gov_goods_receipts + gov_goods_receipt_items
  Auto-generates: InventoryMovement records (IN type)
  Projects to Snipe-IT: increment qty via atomic DB queries

4.2 Goods Issue (Outbound Stock)
--------------------------------
  Path: /gov-store/operations/issues/create
  
  Storekeeper issues goods directly (no approval workflow).
  Creates: gov_goods_issues + gov_goods_issue_items
  Auto-generates: InventoryMovement records (OUT type)
  Projects to Snipe-IT: decrement qty via atomic DB queries

4.3 Stock Adjustment (Corrections)
-----------------------------------
  Path: Not yet implemented in UI
  
  For physical count corrections, damage, loss, expired items.
  Creates: gov_stock_adjustments + gov_stock_adjustment_items
  Auto-generates: InventoryMovement records (IN or OUT type)

4.4 Stock Card / Kardex (Audit Trail)
--------------------------------------
  Path: /gov-store/operations/kardex/{type}/{id}
  
  Immutable ledger showing all movements for a specific item.
  Columns: Date, Reference Document, Operator, IN(+), OUT(-), Running Balance

4.5 Key Models
--------------
  GoodsReceipt      -> Inbound document header
  GoodsReceiptItem  -> Inbound line items
  GoodsIssue        -> Outbound document header
  GoodsIssueItem    -> Outbound line items
  StockAdjustment   -> Correction document header
  StockAdjustmentItem -> Correction line items
  InventoryMovement -> Immutable audit ledger (one record per item per doc)

5. INTEGRATION POINTS (CURRENT STATE)
================================================================================

5.1 Shared Infrastructure
-------------------------
  Both packages share:
  - GovStore\TenantScope\Contexts\TenantContext (tenant scoping)
  - MinistryLocationScope (global scope on models)
  - Similar adapter pattern (store-operations has StockableInterface,
    custom-requests has RequestableInterface)
  - Factory pattern for polymorphic type resolution

5.2 Adapter Comparison (Parallel Implementations)
-------------------------------------------------
  store-operations:                    custom-requests:
  StockableInterface                   RequestableInterface
  ConsumableAdapter                    ConsumableAdapter
  AccessoryAdapter                     AccessoryAdapter
  ComponentAdapter                     AssetAdapter
                                     + ConsumableAdapter
  
  Key difference:
  - store-operations adapters use atomic DB::table()->increment/decrement()
    to bypass Snipe-IT observers (prevents side effects)
  - custom-requests adapters call Snipe-IT's native ->checkout() which triggers
    Snipe-IT's own observers and logging

5.3 The Fulfillment Gap (CRITICAL)
----------------------------------
  When a Service Request is fulfilled in custom-requests:
  
  Current flow:
    GovFulfillmentController::process()
      -> FulfillmentService::issueItems()
        -> RequestableFactory::make($type, $id)
          -> Adapter->checkout($user, $storekeeper, $qty, $notes)
            -> Snipe-IT native checkout (decrements qty directly)
  
  What's MISSING:
    NO Goods Issue document created in store-operations
    NO InventoryMovement record in the immutable ledger
    NO audit trail linking stock outflow to the approved Service Request
  
  Result: Stock leaves the building but no formal Goods Issue document exists.

5.4 Duplicate Outbound Flow
---------------------------
  Two separate UIs can reduce stock:
  
  A) /gov-requests/fulfillment/{id} (custom-requests)
     - Linked to approved Service Request
     - Has approval history, substitutions, timeline
     - Updates Snipe-IT directly
  
  B) /gov-store/operations/issues/create (store-operations)
     - Standalone Goods Issue form
     - No approval workflow
     - Creates formal audit document
     - Projects to Snipe-IT via atomic queries
  
  USER CONFUSION RISK: A storekeeper may not know which UI to use.

6. CRITICAL GAPS & RISKS
================================================================================

6.1 AUDIT GAP (High Severity)
-----------------------------
  Problem: Service Request fulfillments bypass the immutable ledger.
  
  Scenario:
    1. Employee submits SR-2026-000123 for 10 laptops
    2. Approver approves
    3. Storekeeper fulfills via /gov-requests/fulfillment
    4. Snipe-IT qty decreases by 10
  
  What's missing:
    - No gov_goods_issues record exists
    - No InventoryMovement records exist
    - Kardex for those laptops shows no OUT entry
    - Auditors cannot trace stock outflow to the approved request
  
  Impact: Financial audit failure. Stock leaves without formal documentation.

6.2 DUAL ENTRY RISK (Medium Severity)
-------------------------------------
  Problem: Storekeeper might use both UIs for the same stock movement.
  
  Scenario:
    1. Fulfillment already issued 10 items via /gov-requests/fulfillment
    2. Storekeeper creates Goods Issue at /gov-store/operations/issues/create
       for the same 10 items
    3. Snipe-IT qty decremented twice = -10 (negative stock)
  
  Mitigation: store-operations has pre-validation, but this is a procedural
  issue, not technical. Users need clear guidance on when to use which UI.

6.3 LEGACY MODEL CONFUSION (Low-Medium Severity)
-------------------------------------------------
  Problem: ItemRequest model exists alongside Request model.
  
  - ItemRequest: single-item, simple pending status, no lifecycle
  - Request: multi-item, full approval/fulfillment workflow
  
  Both can be triggered from the same UI (item page "request" button).
  Users won't know which path their request takes.

6.4 ADAPTER PATTERN DUPLICATION (Low Severity)
-----------------------------------------------
  Both packages implement nearly identical adapter patterns:
  
  - StockableInterface vs RequestableInterface (same purpose, different names)
  - ConsumableAdapter/AccessoryAdapter in both packages
  - StockableFactory vs RequestableFactory
  
  This creates maintenance burden and potential inconsistency.

6.5 NO CROSS-REFERENCE BETWEEN PACKAGES
---------------------------------------
  Problem: Service Requests and Goods Issues are completely disconnected.
  
  A fulfilled Service Request should automatically create a corresponding
  Goods Issue document in store-operations, linking:
    - gov_goods_issues.issue_no = GI-2026-XXXXXX
    - gov_goods_issues.reference_type = GovStore\CustomRequests\Models\Request
    - gov_goods_issues.reference_id = <service_request_id>
  
  Currently this link does not exist.

  3. "Why are there two request models (ItemRequest and Request)?"

  4. "Does fulfilling a Service Request create an audit document?"

8. RECOMMENDED ARCHITECTURE
================================================================================

8.1 Clear Separation of Concerns
---------------------------------
  
  +---------------------------+-----------------------------------+
  | custom-requests           | store-operations                  |
  |                           |                                   |
  | APPROVAL WORKFLOW         | INVENTORY ACCOUNTING              |
  |                           |                                   |
  | - Employee submits        | - Goods Receipt (inbound)         |
  | - Approver reviews        | - Goods Issue (outbound)          |
  | - Basket aggregation      | - Stock Adjustments               |
  | - Policy-based routing    | - Immutable Movement Ledger       |
  | - Product substitution    | - Kardex / Stock Card             |
  |                           |                                   |
  | OUTPUT: Approved Request  | INPUT: Formal Documents           |
  +---------------------------+-----------------------------------+

8.2 Required Integration: Fulfillment -> store-operations
---------------------------------------------------------
  
  When FulfillmentService issues items, it MUST ALSO create a Goods Issue
  document in store-operations:
  
  Current flow (INCOMPLETE):
    FulfillmentService::issueItems()
      -> Adapter->checkout() [updates Snipe-IT]
  
  Required flow (COMPLETE):
    FulfillmentService::issueItems()
      -> GoodsIssueService::createFromFulfillment($serviceRequest, $items)
        -> Creates gov_goods_issues header
          -> Creates gov_goods_issue_items lines
            -> Creates InventoryMovement records (OUT)
              -> Fires Projection Engine
                -> Adapter->decrementQuantity() [updates Snipe-IT]
  
  This ensures:
    - Every stock outflow has a formal Goods Issue document
    - The Goods Issue references the Service Request (polymorphic link)
    - Kardex shows complete audit trail
    - Auditors can trace from ledger -> goods issue -> service request

8.3 Recommended Changes to custom-requests
-------------------------------------------
  
  A) Deprecate ItemRequest model
     - Mark as legacy, redirect all new submissions to Request model
     - Show deprecation notice in UI
  
  B) FulfillmentService must call store-operations
     - Add dependency on GoodsIssueService or create integration layer
     - Every fulfillment creates a corresponding Goods Issue document
     - Link via polymorphic reference:
       gov_goods_issues.reference_type = GovStore\CustomRequests\Models\Request
       gov_goods_issues.reference_id = <request_id>
  
  C) Remove duplicate adapter logic
     - Either share adapters between packages or keep them separate with
       clear documentation of their different purposes

8.4 Recommended Changes to store-operations
--------------------------------------------
  
  A) Add polymorphic reference fields to GoodsIssue model
     - reference_type (nullable) -> links to Service Request
     - reference_id (nullable)   -> links to Service Request ID
  
  B) Add createFromFulfillment() method to GoodsIssueService
     - Accepts a ServiceRequest and issue quantities
     - Creates Goods Issue header with auto-generated number
     - Maps each RequestItem to a GoodsIssueItem
     - Sets reference_type/reference_id for traceability
  
  C) Clarify UI guidance
     - Add tooltip/banner: "Use this form for ad-hoc stock issues.
       For approved employee requests, use the Fulfillment queue."

8.5 Unified Navigation Recommendation
--------------------------------------
  
  Both packages should be accessible from a single "Gov Store" menu in the
  Snipe-IT sidebar, organized by role:
  
  Gov Store Operations
  ├── My Requests          -> custom-requests (employee view)
  ├── Approval Queue       -> custom-requests (approver view)
  ├── Fulfillment Queue    -> custom-requests (storekeeper view)
  ├── Receive Goods        -> store-operations (inbound)
  ├── Issue Stock          -> store-operations (ad-hoc outbound)
  ├── Stock Cards          -> store-operations (kardex search)
  └── Settings
      ├── Approval Policies -> custom-requests
      └── Office Locations  -> office-membership package
